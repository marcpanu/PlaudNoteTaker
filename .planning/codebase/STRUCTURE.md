## Directory Purposes

**`src/`:**
- Purpose: All TypeScript source code
- Contains: Feature modules organized by domain, CLI handlers, utilities
- Key files: `index.ts` (entrypoint), `pipeline.ts` (main logic), `config.ts` (env loading)

**`src/cli/`:**
- Purpose: User-facing command implementations
- Contains: Five command handlers, each exported as `run<Command>()`
- Entry point: Each is imported and dispatched from `src/index.ts`
- Pattern: Handlers load config, validate inputs, call domain logic, handle errors and display output

**`src/plaud/`:**
- Purpose: Plaud device API communication
- Contains: TypeScript types for API responses and HTTP client with retry logic
- Key pattern: All Plaud API calls go through `PlaudClient` singleton; retry logic built in for rate limiting (429) and server errors (5xx)

**`src/transcription/`:**
- Purpose: Speech-to-text and speaker diarization
- Contains: Only AssemblyAI integration (single provider)
- Key exports: `transcribeAudio()` returns `Utterance[]` with speaker labels
- Pattern: Upload audio, submit job, poll status until complete or error

**`src/summarization/`:**
- Purpose: Generate structured meeting notes from transcripts
- Contains: Only Gemini API integration (single provider)
- Key logic: Template population, folder routing extraction, markdown parsing

**`src/speakers/`:**
- Purpose: Speaker identification and enrollment
- Contains: Picovoice Eagle integration and profile storage
- Key functions: `recognizeSpeakers()`, `enrollSpeaker()`, profile serialization helpers
- State: Profiles stored as base64-encoded ArrayBuffers in JSON

**`src/notes/`:**
- Purpose: Markdown generation and Obsidian vault integration
- Contains:
  - `writer.ts`: Full note building, filename sanitization, label parsing/application
  - `template.ts`: Template loading with fallback to DEFAULT_TEMPLATE
  - `vault.ts`: Directory tree scanning for LLM context

**`src/audio.ts`:**
- Purpose: Audio format conversion
- Contains: Two functions — WAV conversion (Plaud OGG → transcription-friendly WAV) and PCM decoding (any format → Int16 raw)
- Dependencies: ffmpeg (external process)
- Temporary files: Created in OS tmpdir, cleaned up after use

**`data/`:**
- Purpose: Runtime application data (not version controlled)
- Created by: `plaud init` command
- Contains: Four JSON files tracking state across command invocations
- Pattern: Simple read/write operations, no transaction handling

**`dist/`:**
- Purpose: Compiled JavaScript output
- Created by: `npm run build` (runs `tsc`)
- Usage: `dist/index.js` executed by `npm start` or `node dist/index.js`
- Git: Ignored (not committed)

## Key File Locations

**Entry Points:**
- `src/index.ts`: Main CLI dispatcher, handles `process.argv`, routes to command handlers
- `dist/index.js`: Compiled entrypoint (has shebang, runs as `plaud` command)

**Configuration:**
- `.env`: Runtime config loaded by `src/config.ts` at startup (created by `plaud init`)
- `.env.example`: Template showing all required/optional variables
- `src/config.ts`: Validates and exports `Config` type

**Core Logic:**
- `src/pipeline.ts`: Main recording processing loop (9 steps: download → transcribe → recognize → summarize → write)
- `src/cli/start.ts`: Polling loop that fetches new recordings and calls pipeline

**State:**
- `src/state.ts`: All persistent state operations (JSON file I/O)
- `data/processed-recordings.json`: Tracks which recordings have been processed (prevents duplicates)
- `data/recording-meta.json`: Stores utterance metadata for post-processing via `plaud label`
- `data/speaker-profiles.json`: Enrolled speaker profiles (base64-encoded)

**Testing:**
- No dedicated test files. Manual testing via `plaud test` command (validates API connections).

## Naming Conventions

**Files:**
- Modules: lowercase with hyphens only in domain folders (`src/plaud/client.ts`, not `src/plaud/plaud-client.ts`)
- Exports: `src/cli/*.ts` export `run<Command>()` (e.g., `runInit`, `runStart`, `runLabel`)
- Classes: CamelCase (e.g., `PlaudClient`, `EagleProfiler` from SDK)
- JSON state files: kebab-case (`processed-recordings.json`, `recording-meta.json`)

**Directories:**
- Feature domains: lowercase plural or singular depending on convention
  - `src/cli/` (multiple commands)
  - `src/plaud/`, `src/transcription/`, `src/summarization/` (single domain each)
  - `src/speakers/`, `src/notes/` (plural, multiple concerns)

**Functions:**
- camelCase for all functions
- Descriptive names: `transcribeAudio()`, `recognizeSpeakers()`, `buildTranscriptText()`, `parseUnknownSpeakers()`
- Verbs first: `load`, `save`, `build`, `extract`, `recognize`, `enroll`

**Variables & Constants:**
- camelCase for all variables
- UPPERCASE only for module-level constants that aren't exported: `const MAX_RETRIES = 3`, `const SCORE_THRESHOLD = 0.3`
- Booleans start with `is` or `has`: `isTrash`, `haRemaining`, `isRetryable`

**Types:**
- PascalCase for interfaces and types: `Utterance`, `PlaudRecording`, `SummaryResult`, `Config`
- Descriptive: `PlaudRecordingsResponse`, `PlaudTempUrlResponse` (includes API response context)

## Where to Add New Code

**New Feature (e.g., speaker enrollment overhaul):**
- Primary code: `src/speakers/` (add functions to `eagle.ts` or new module)
- Domain logic: Create in appropriate domain folder (`src/cli/`, `src/transcription/`, etc.)
- Utilities: If reusable across features, add to `src/` root (like `audio.ts`)
- State: If needs persistence, extend `src/state.ts` with new JSON file + load/save functions

**New Command (e.g., `plaud export`):**
- Handler: Create `src/cli/export.ts` exporting `runExport()`
- Register: Import in `src/index.ts`, add to command switch statement
- Config: Add any new env vars to `.env.example` and `src/config.ts`

**New External Service Integration (e.g., Anthropic API instead of Gemini):**
- Create new folder: `src/summarization/anthropic.ts` (not a new folder, reuse `summarization/`)
- Or create folder: `src/api/` if integrating completely different service class
- Export same interface as existing implementation for drop-in replacement
- Update `src/config.ts` to load new API key
- Update `src/pipeline.ts` to use new service

**Utilities/Helpers:**
- Shared across commands/domains: `src/` root (e.g., `audio.ts`, `log.ts`, `config.ts`)
- Domain-specific helpers: Domain folder (e.g., speaker label parsing in `src/speakers/profiles.ts`)

## Special Directories

**`data/`:**
- Purpose: Application runtime data
- Generated: Yes (created by `plaud init`, populated by pipeline)
- Committed: No (in `.gitignore`)
- Contents: JSON files, user's private speaker profiles and recording metadata

**`dist/`:**
- Purpose: Compiled TypeScript output
- Generated: Yes (by `npm run build`)
- Committed: No (in `.gitignore`)
- Contents: `.js` files compiled from `src/`, runnable with Node.js

**`.claude/`:**
- Purpose: Claude IDE/agent workspace data
- Generated: Yes (by Claude tools)
- Committed: No
- Contents: Internal metadata

**`.planning/`:**
- Purpose: GSD analysis and planning documents
- Generated: Yes (by `/gsd` commands)
- Committed: Planned (for team reference)
- Contents: `ARCHITECTURE.md`, `STRUCTURE.md`, `CONVENTIONS.md`, etc.
